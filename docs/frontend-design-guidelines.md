# Frontend Design Guidelines

本仓库的 userscript UI 默认按产品工具界面处理：设计服务于任务，优先清晰、可靠、低干扰，而不是营销页式视觉冲击。

## Product UI Register

- 使用系统字体栈，保持浏览器原生工具感。
- 结构优先于装饰。设置面板、浮动控件、表单和状态信息要让用户快速判断当前配置。
- 熟悉的控件就是功能：按钮、输入框、单选、复选、状态提示不要重新发明交互方式。
- 动效只表达状态变化，常规过渡保持在 150ms 到 250ms。

## Color

- 新增 UI 色值优先使用 `oklch()`。
- 不使用纯 `#000` 或 `#fff`。中性色应轻微偏向产品主色相，避免刺眼。
- 默认采用 restrained 策略：大面积使用有轻微色相的中性色，一个主强调色只用于主操作、选中态、focus 和状态提示。
- 危险操作使用单独的低饱和警示色，不和主强调色混用。

## Layout

- 弹窗和浮动面板保持稳定尺寸，避免 hover、文本或状态切换造成明显布局跳动。
- 控件间距要有层级：区块之间更宽，相关控件之间更紧。
- 避免嵌套卡片。状态盒、选项块可以有边框和浅底色，但不要把每个元素都包装成装饰卡片。
- 移动端要保证表单控件可点击、文本不挤压、按钮可以自然换行。

## Components

- 每个交互控件至少覆盖 default、hover、focus-visible、active、disabled 状态。
- 图标按钮使用真实按钮元素和 `aria-label`。
- 表单选项的选中态要同时通过控件本身和容器状态表达，例如 `accent-color` 加边框/背景变化。
- 注入 CSS 必须限定在脚本 root 下，除非用户明确启用了影响宿主页面的能力。

## Banned Patterns

- 不使用渐变文字。
- 不把粗侧边框作为卡片或提示的装饰。
- 不把 glassmorphism 作为默认风格。模糊背景只在有明确理由时使用。
- 不使用重复的图标加标题加说明卡片网格来填充设置界面。
- 不用模态框作为所有问题的默认答案；本仓库已有设置入口可以复用时，优先复用现有面板。
